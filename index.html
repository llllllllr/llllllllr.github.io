JavaScript事件循环机制

发表于 2020-10-07 | 分类于 font-end

v8 — call stack and a heap

go to the task queue whenever the stack is clear

总结：

  • 所有的代码都要通过函数调用栈中调用执行
  • 当遇到前文中提到的APIs，会交给浏览器内核的其它模块进行处理
  • 任务队列中存放的是回调函数
  • 等到调用栈中的task执行完之后再回去执行任务队列中的task

例子

  1. 首先i=0时，满足条件，执行栈执行循环体里面的代码，发现是setTimeout，将其出栈之后把延时执行的函数交给Timer模块进行处理。
  2. 当i=1,2,3,4时，均满足条件，情况和i=0时相同，因此timer模块里面有5个相同的延时执行的函数。
  3. 当i=5的时候，不满足条件，因此for循环结束,console.log(new Date, i)入栈，此时的i已经变成了5。因此输出5。
  4. 此时1s已经过去，timer模块将5个回调函数按照注册的顺序返回给任务队列。
  5. 执行引擎去执行任务队列中的函数，5个function依次入栈执行之后再出栈，此时的i已经变成了5。因此几乎同时输出5个5。
  6. 因此等待的1s的时间其实只有输出第一个5之后需要等待1s，这1s的时间是timer模块需要等到的规定的1s时间之后才将回调函数交给任务队列。等执行栈执行完毕之后再去执行任务对列中的5个回调函数。这期间是不需要等待1s的。因此输出的状态就是：5 -> 5,5,5,5,5，即第1个 5 直接输出，1s之后，输出 5个5；

宏任务和微任务

前面我们介绍过，在一个事件循环中，异步事件返回结果后会被放到一个任务队列中。然而，根据这个异步事件的类型，这个事件实际上会被对应的宏任务队列或者微任务队列中去。并且在当前执行栈为空的时候，主线程会 查看微任务队列是否有事件存在。如果不存在，那么再去宏任务队列中取出一个事件并把对应的回到加入当前执行栈；如果存在，则会依次执行队列中事件对应的回调，直到微任务队列为空，然后去宏任务队列中取出最前面的一个事件，把对应的回调加入当前执行栈…如此反复，进入循环。

我们只需记住当当前执行栈执行完毕时会立刻先处理所有微任务队列中的事件，然后再去宏任务队列中取出一个事件。同一次事件循环中，微任务永远在宏任务之前执行。

浏览器渲染页面过程

浏览器

内核

  • Trident内核，IE
  • Gecko内核, FireFox
  • Webkit内核， Safari 以前的chrome
  • Pressto, Opera
  • Blink, Chrome
  • Chrome JS引擎，V8

工作流程

  • 浏览器解析的文件
    • HTML
    • CSS
    • JavaScript
  • 解析完成后，浏览器引擎通过DOM TREE和 CSS Rule Tree 来构造 Renderint Tree
  • 最后通过调用操作系统的Native GUI 的 API绘制
构建DOM

字节数据 -> 字符串 -> Token -> Node -> DOM

构建CSSOM

字节数据 -> 字符串 -> Token -> Node -> CSSOM

  • css匹配HTML元素 从右往左解析

    如果正向解析，例如【div div p em】，我们首先就要检查当前元素到html的整条路经，找到嘴上的div，再往下找，如果遇到不匹配就必须回到最上层的那个div，往下再去匹配选择器中的第一个div，回溯若干次才能确定匹配与否。

生成渲染树

渲染树只会包括需要显示的节点和这些节点的样式信息，如果某个节点是 display:none， 那么就不会在渲染树中显示。

布局与绘制

当浏览器生成渲染树以后，就会根据渲染树来进行布局（也可以叫做回流），这一阶段浏览器要做的事情就是定位元素的确切位置和大小，通常这一行为也被称为“自动重排”。

window.onload 和 DOMContentLoaded的区别

在Js中DOMContentLoaded方法时在HTML文档被完全的加载和解析之后才会触发的事件，他并不需要等到（样式表/图像/子框架）加载之后再进行。再看load事件（onload事件），用于检测一个加载完全的页面。

DOM完整的解析过程：

  • 解析HTML结构
  • 加载外部脚本和样式表文件
  • 解析并执行脚本代码 // js
  • DOM树构建完成 // DOMContentLoaded
  • 加载图片等外部文件
  • 页面加载完成 // loaded

当onload事件触发时，页面上所有的DOM,样式表，图片，flash都已经加载完成了

当DOMContentloaded事件触发时，仅当DOM加载完成，不包括样式表，图片，flash

其他问题

渲染过程中遇到JS文件怎么处理

JavaScript的加载，解析和执行会阻塞DOM的构建，也就是说，在构建DOM时，HTML解析器若遇到了JavaScript，那么它会暂停构建DOM，将控制权交给JavaScript引擎，等JavaScript引擎运行完毕，浏览器再从中断的地方恢复DOM构建。

这时候要提到async和defer

  • 两者都会并行下载，不会影响页面的解析。
  • defer会按照顺序在DOMContentLoaded前按照页面出现的顺序依次执行。
  • async则是下载完立即执行。

对于一个普通的script标签

  1. 停止解析document
  2. 请求a.js
  3. 执行a.js中的脚本
  4. 继续解析document

defer

  1. 不阻止解析document，并行下载js
  2. 即使下载完js,仍继续解析document
  3. 按照页面中出现的顺序，在其他同步脚本执行后，DOMContendLoaded事件前依次执行下载的js

async

  1. 不阻止解析document,并行下载js
  2. 脚本下载完立即执行（两者的执行顺序不确定，执行阶段不确定，可能在DOMContentloaded之后，适用于无依赖脚本）
  3. 即使可以在DOMContentLoaded之后执行，也一定会在onload事件触发之前执行。

总结

  1. 两者都不会阻止document的解析
  2. defer会在DOMContentLoaded前依次执行
  3. async则是下载完立即执行，不一定是在DOMContentLoaded前
  4. async因为顺序无关，所以很适合Google Analytics这种无依赖脚本。

参考：https://github.com/xiaoyu2er/blog/issues/8

Js文件不只是阻塞DOM的构建，而且还会导致CSSOM也阻塞DOM的构建。

原本DOM和CSSOM的阻塞是互不影响的，井水不犯河水，但是一旦引入了JavaScript，CSSOM也开始阻塞DOM执行，等CSSOM执行完毕之后，DOM再恢复DOM构建。

因为：

  • JavaScript不只是可以改DOM，它还可以更改样式，也就是可以更改CSSOM
  • 不完整的CSSOM是无法使用的，但JavaScript中像访问CSSOM并更改它，那么在执行JavaScript时，必须要拿到完整CSSOM。所以弱国浏览器上文完成CSSOM的下载和构建，而我们却想在此时运行脚本，那么浏览器将延迟脚本执行和DOM构建，直到它完成CSSOM的下载和构建。也就是说，在这种情况下，浏览器会先下载和构建CSSOM，然后再执行JavaScript，最后再继续构建DOM

回流和重绘

  • 重绘：当Render Tree中的一些元素需要更新属性，而这些属性只是影响元素的外观、风格，而不会影响布局的，例如background-color
  • 回流：当render tree中的一部分因为元素的规模尺寸、布局、隐藏等改变而需要重新构建
常见的引起回流属性和方法

重点是改变元素的位置，大小的操作，都会触发回流

  • 添加/删除可见的DOM元素
  • 元素尺寸改变，边距、填充、边框、宽度和高度
  • 内容变化，用户在input框里输入文字
  • 浏览器窗口尺寸改变—resize事件发生
  • 计算offsetWidth和offsetHeight属性
  • 设置style属性的值

改善：

  • 使用transform替代top
  • 使用visibility替换display:none，因为前者只会引起重绘，而后者引发回流，因为改变了布局
  • 不要把节点的属性放在一个循环里当成循环里面的变量

浏览器中的进程和线程

进程和线程

  1. 一个线程只能属于一个进程，一个进程可以有多个线程，但至少有一个线程
  2. 资源分配给进程，同一个进程的所有线程共享该进程的所有资源
  3. CPU分配给线程，真正在CPU上运行的是线程
  4. 线程是进程内的一个执行的执行单元，也是进程内的可调度实体

进程间通信

（1）管道（pipe）及有名管道（named pipe）：管道可用于具有亲缘关系的父子进程间的通信，有名管道除了具有管道所具有的功能外，它还允许无亲缘关系进程间的通信。

（2）信号（signal）：信号是在软件层次上对中断机制的一种模拟，它是比较复杂的通信方式，用于通知进程有某事件发生，一个进程收到一个信号与处理器收到一个中断请求效果上可以说是一致的。

（3）消息队列（message queue）：消息队列是消息的链接表，它克服了上两种通信方式中信号量有限的缺点，具有写权限得进程可以按照一定得规则向消息队列中添加新信息；对消息队列有读权限得进程则可以从消息队列中读取信息。

（4）共享内存（shared memory）：可以说这是最有用的进程间通信方式。它使得多个进程可以访问同一块内存空间，不同进程可以及时看到对方进程中对共享内存中数据得更新。这种方式需要依靠某种同步操作，如互斥锁和信号量等。

（5）信号量（semaphore）：主要作为进程之间及同一种进程的不同线程之间得同步和互斥手段。

（6）套接字（socket）：这是一种更为一般得进程间通信机制，它可用于网络中不同机器之间的进程间通信，应用非常广泛。进程和线程

在chrome浏览器中，打开一个网页相当于起了一个进程。

  • 进程之间不共享资源和地址空间，所以不会存在太多的安全问题，而由于多个线程共享这相同的地址空间和资源，所以会存在线程之间有可能会恶意修改或者非授权数据等复杂的安全问题

浏览器内核的复杂问题

  • GUI渲染线程

    • 负责渲染浏览器界面HTML元素，当界面需要重绘或回流再执行。在Js脚本运行 期间，这个线程是挂起状态。

  • JavaScript引擎线程

    • Js内核，主要负责JavaScript脚本程序，例如V8引擎。

    ⭐注意：上面两个线程是互斥的

  • 定时触发器线程

    • 浏览器定时计数器不是由JavaScript引擎计数的，因为JavaScript引擎是单线程的，如果处于阻塞线程状态就会影响计时的准确性。

  • 事件触发线程

    • 当一个事件被触发时该线程会把事件添加到待处理队列的对位，等待JS引擎的处理。

  • 异步Http请求线程

    • XMLHttpRequest在连接后是通过浏览器新开一个县城请求，将检测到状态变更时，如果设置有回调函数，异步线程就产生状态变更事件放到JavaScript引擎的处理队列中处理。

从输入url到页面加载发生了什么

发表于 2020-10-06 | 分类于 front-end

从输入URL到页面加载发生了什么

概述

  • DNS解析
  • TCP连接
  • 发送HTTP请求
  • 服务器处理请求并返回HTTP报文
  • 浏览器解析渲染页面
  • 连接结束
DNS解析

  1. 先在本地host文件查找域名对应的ip地址

  2. 如果host文件没有的话，会查找本地DNS解析器缓存，是否有这个网址映射关系，如果有，直接返回，完成域名解析

  3. 如果都没有，找TCP/IP参数中设置的首选DNS服务器，也就是本地DNS服务器，本地域名服务器如果包含在本地配置区域资源中，就返回解析结果，完成域名解析

  4. 如果要查询的域名不是由本地域名服务器解析，但是服务器已经缓存了这个网址的映射关系，就调用这个IP地址映射，完成域名解析

  5. 如果本地域名服务器和缓存解析都失效，就根据本地DNS服务器的设置(是否设置转发器)进行查询，如果没有用转发模式，本地DNS就会把请求发到13台根域名服务器，根域名服务器所受到请求后会判断这个域名（.com）是谁来授权管理，并且会返回一个负责该顶级域名服务器的一个IP。

  6. 本地DNS服务器拿到这个IP信息之后会联系这台负责这个域的服务器，这台负责.com域的服务器收到请求之后，如果自己无法解析，就会找一个管理.com域的下一级DNS服务器地址（qq.com）给本地域名服务器，本地域名服务器拿到之后又会重复上面的动作，继续进行查询，知道找到www.qq.com的主机。

    主机向本地域名服务器的查询一般采用递归查询(交给下一任)

    本地域名服务器向根域名服务器的查询属于迭代查询（下一任收到之后返回下一个域的ip地址）

TCP连接

tcp是传输层协议，http是网络层协议

三次握手

  1. 第一次握手
    • 客户端发送一个SYN标记的包，初始序列号为x，发送完毕后客户端进入SYN_SEND状态
  2. 第二次握手
    • 服务端发送ACK应答，ACK = x+1，表示收到了客户端发送的SYN包，并且带有自己的序列号seq = y，表示让客户端确认是否能收到
  3. 第三次握手
    • 客户端再次发送确认包，表示确认收到服务器的包（服务器发送的seq值+1）

四次挥手

  1. 第一次挥手
    • 客户端发送一个FIN标记的包，告诉服务器要关闭连接，表示自己不用再发送数据了，但是还可以接受数据。发送完成后，客户端进入FIN_WAIT_1状态
  2. 第二次挥手
    • 服务器发送一个ACK确认包，告诉客户端自己收到关闭的请求，但是还没准备好。发送完成后，客户端进入CLOSE_WAIT状态，客户端接收到这个包之后，进入FIN_WAIT_2，等待服务器关闭连接
  3. 第三次挥手
    • 服务器准备好之后，发送FIN标记的包，发送完成后，服务端进入LAST_ACK状态，等待客户端进行确认
  4. 第四次挥手
    • 客户端接收到服务端的关闭请求，再发送ACK标记的确认包，进入TIME_WAIT状态，等待服务端可能请求重传的ACK包。
    • 服务端接收到ACK包后，关闭连接
    • 客户端在等待两个最大段生命周期后，没有接收到确认的ACK包，认为服务器已经关闭连接，自己也关闭连接，进入CLOSE状态

TCP可靠传输：

  • TCP给发送的每一个包进行编号，接收方对数据包进行排序，把有序数据传送给应用层
  • 校验和
  • 流量控制：TCP的每一方都有固定大小的缓冲空间，TCP的接收端只允许发送端发送接收端缓冲区能接纳的数据，当接收方来不及处理发送方的书库，能提示发送方降低发送速率，防止包丢失。TCP使用的流量控制协议是可变大小的滑动窗口协议。
  • 拥塞控制：当网络拥塞时，减少数据的发送
  • 停止等待协议：每发送完一个分组就停止发送，等待对方确认（超时重传机制）

HTTP请求头

  1. Accept
    • Accept: text/html 浏览器可以接收服务器回发的类型为text/html
    • Accept: */* 浏览器可以处理所有类型
  2. Accept-Encoding
    • gzip，浏览器申明自己接收的压缩方法（是否支持压缩，支持什么压缩方法）
  3. Accept-Language
    • zh-CN 申明自己接收的语言
  4. Connection
    • keep-alive: 当一个网页打开完成后，客户端和服务器之间用于传输HTTP数据的TCP连接不会关闭，如果客户端再次访问这个服务器上的网页，会继续使用这一条已经建立的连接
    • close，表明一个Request完成后，客户端和服务器之间用于传输HTTP数据的TCP连接会关闭，当客户端再次发送请求，需要重新建立TCP连接
  5. Host
    • 请求资源的主机和端口号，通常从HTTP URL中提取出来
  6. Referer
    • 当浏览器向web服务器发送请求的时候，一般会带上Referer，告诉服务器我是从哪个页面链接过来的，服务器借此可以获得一些信息
  7. User-Agent
    • 告诉http服务器，客户端使用的操作系统和浏览器的版本和名称
  8. Cache-Control
  9. Cookie
  10. Range
    • Range:bytes=0-5,指定第一个字节和最后一个字节的位置，用于告诉服务器自己想取对象的哪一部分
http缓存

⭐强缓存：浏览器不会向服务器发送任何请求，直接从本地缓存中读取文件并返回Status Code: 200 ok

⭐协商缓存：向服务器发送请求，服务器会根据这个请求的request header的一些参数来判断是否命中协商缓存，如果命中，则返回304状态码，并带上新的response header通知浏览器从缓存中读取资源

缓存header参数

🏃‍强缓存

  • Expires（http1.0）： 过期时间

  • Cache-Control（http1.1）： 假如值为300，请求正确返回时间的300秒（5分钟）之内再次加载资源就会命中强缓存

    1. max-age
    2. s-maxage：和max-age一样，但针对代理服务器缓存
    3. public: 响应可被任何缓存区缓存
    4. private: 只能针对个人用户，不能被代理服务器缓存
    5. no-cache: 强制客户端直接向服务器发送请求，也就是说每次请求都必须向服务器发送，如果没有变更的话会返回304
    6. no-store: 禁止一切缓存

🏃‍协商缓存

  • Etag/If-None-Match(http1.1)

    Etag由服务器生成返回给前端，由文件的索引，大小，最后修改时间等进行hash得到的

    当资源过期时，浏览器发现响应头里有Etag,再次向服务器请求时会带上请求头If-None-Match，服务器收到请求进行对比

  • Last-Modifed/If-Modified-Since(http1.0)

    Last-Modified: 浏览器向服务器发送资源的最后修改时间

    当资源过期时，浏览器判断Cache-Control的max-age标识，发现响应头具有Last-Modified声明，则再次向服务器请求时带上if-modified-since,表示请求事件。服务器收到请求后发现if-modified-since则与被请求资源最后修改时间进行对比

防抖

防抖的原理是不管你在一段时间内如何不停的触发事件，只要设置了防抖，则只在触发n秒后才执行。如果我们在一个事件触发的n秒内又触发了相同的事件，那我们便以新的事件时间为标准，n秒之后再执行。

应用场景

两个条件:
1,如果客户连续的操作会导致频繁的事件回调(可能引起页面卡顿).
2,客户只关心”最后一次”操作(也可以理解为停止连续操作后)所返回的结果.
例如:

  • 输入搜索联想，用户在不断输入值时，用防抖来节约请求资源。
  • 按钮点击:收藏,点赞,心标等

不想等到停止触发事件后才执行对应代码，而是希望立马执行，等到停止执行n秒后，又再重新触发执行。

小tips:

所以 apply 和 call 的区别是 call 方法接受的是若干个参数列表，而 apply 接收的是一个包含多个参数的数组。

节流

节流的原理是不管你在一段时间内如何不停地触发事件，只要设置了节流，就会每隔一段时间执行一次。

  1. 时间戳方式:通过闭包保存上一次的时间戳,然后与事件触发的时间戳比较.如果大于规定时间,则执行回调.否则就什么都不处理.
  • 特点:一般第一次会立即执行，之后连续频繁地触发事件，也是超过了规定时间才会执行一次。最后一次触发事件，也不会执行(说明:如果你最后一次触发时间大于规定时间,这样就算不上连续频繁触发了).
  1. 定时器方式:原理与防抖类似.通过闭包保存上一次定时器状态.然后事件触发时,如果定时器为null(即代表此时间隔已经大于规定时间),则设置新的定时器.到时间后执行回调函数,并将定时器置为null.
  • 特点:当第一次触发事件时，不会立即执行函数，到了规定时间后才会执行。 之后连续频繁地触发事件，也是到了规定时间才会执行一次(因为定时器)。当最后一次停止触发后，由于定时器的延时，还会执行一次回调函数(那也是上一次成功成功触发执行的回调,而不是你最后一次触发产生的)。一句话总结就是延时回调,你能看到的回调都是上次成功触发产生的,而不是你此刻触发产生的.

说明: 这两者最大的区别:是时间戳版的函数触发是在规定时间开始的时候，而定时器版的函数触发是在规定时间结束的时候。

sort algorithm summary

  1. 首先在未排序序列中找到最小（大）元素，存放到排序序列的起始位置
  2. 再从剩余未排序元素中继续寻找最小（大）元素，然后放到已排序序列的末尾。
  3. 重复第二步，直到所有元素均排序完毕。
  1. 将第一待排序序列第一个元素看做一个有序序列，把第二个元素到最后一个元素当成是未排序序列。
  2. 从头到尾依次扫描未排序序列，将扫描到的每个元素插入有序序列的适当位置。（如果待插入的元素与有序序列中的某个元素相等，则将待插入元素插入到相等元素的后面。） // 稳定排序
  1. 选择一个增量序列 t1，t2，……，tk，其中 ti > tj, tk = 1；
  2. 按增量序列个数 k，对序列进行 k 趟排序；
  3. 每趟排序，根据对应的增量 ti，将待排序列分割成若干长度为 m 的子序列，分别对各子表进行直接插入排序。仅增量因子为 1 时，整个序列作为一个表来处理，表长度即为整个序列的长度。

JS--面向对象设计

发表于 2020-06-12 | 分类于 JavaScript

面向对象的程序设计

创建对象

  • 工厂模式

    工厂模式虽然解决了创建多个相似对象的问题，却没有解决对象识别的问题。

  • 构造函数模式

    • 没有显式的创建对象
    • 直接将属性和方法赋值给了this对象
    • 没有return 语句

    🏊‍ new过程

    1. 创建一个新对象

    2. 将构造函数的作用域赋值给新对象（因此this就指向了这个新对象）

    3. 执行构造函数中的代码（为这个新对象添加属性）

    4. 返回新对象

    5. 最后，虽然person1 和person2都有不同的实例，但是这两个对象都有一个constructor(构造函数)属性，该属性指向Person

    • instanceof操作符

    • 将构造函数当作函数

      • 用new操作符构建一个新对象
      • 不使用new操作符 - 属性和方法都被添加给window对象
      • call,bind,apply在某个特殊对象的作用域中调用

    • 构造函数的问题

      • 每个Person实例都会包含一个不同的Function实例
      • 然而，创建两个完成同样任务的Function实例的确没什么必要
      • Person对象的不同实例此时可以共享全局作用域中的同一个sayName()函数，但是问题如下：
        • 全局作用域中定义的函数只能被某个对象调用，全局作用域有点名不副实
        • 对象如果需要定义很多方法，就要定义很多个全局函数，于是自定义的引用类型就丝毫没有封装性可言了

  • 原型模式

    • 我们创建的每个原型都有一个prototype(原型属性)。这个属性是一个指针，指向一个对象，而这个对象的用途是包含可以由特定类型的所有实例共享的属性和方法。

    • prototype:通过调用构造函数二创建的那个对象实例的原型对象

    • 让所有对象实例共享它包含的属性和方法。

  • 新对象还是会具有相同的属性和方法。但是和构造函数模式不同，新对象的这些属性和方法是由所有实例共享的。

  • 理解原型对象

    • 无论什么时候，只要创建了一个新的函数，就会根据一组特定的规则为该函数创建一个prototype属性，这个属性指向原型对象。

    • 而默认情况下，所有原型对象都会自动获得一个constructor属性。

    • constructor这个属性包含一个指向prototype属性所在函数的指针。

    • 比如，Person.prototype.construcor指向Person,通过这个构造函数，我们还可以继续为原型对象添加其它属性和方法。

    • 这个连接存在于实例和构造函数的原型对象之间，而不是实例和构造函数之间。

    • 虽然可以通过实例对象访问保存在原型中的值，但是却不能通过对象实例重写原型中的值。如果我们在实例中添加了一个属性，而该属性与实例原型中的一个属性同名，那么我们就在该实例中创建该属性，该属性会屏蔽原型中的那个属性。

    • 用delete person1.name会删除实例创建的name，恢复原型对象与实例之间的连接。

    • hasOwnProperty()方法可以检测一个属性是存在实例中还是存在原型中。存在实例中返回true.

    • hasPrototypeProperty(实例，属性名)也可，存在原型中返回true.

  • 原型与in操作符

    alert("name" in person1)不管是在原型上还是在实例上访问到，都返回true

  • 更简单的原型语法

    • construcor属性不再指向Person了
      • 每创建一个函数，就会创建它的prototype对象，这个对象自动获得constructor属性。
      • 这里使用的语法，本质上完全重写了默认的prototype对象。constructor属性也变成了新对象的constructor属性，不再指向Person（指向Object构造函数）函数。

  • 原型对象的动态性

    • 我们对原型对象所作的任何修改都能够立刻从实例上反映出来。

    • 如果重写整个原型对象，那么情况就会变得不一样。因为

      1. 调用构造函数时会为实例添加一个指向最初原型的prototype指针

      2. 原型修改为另一个对象就等于切断了构造函数与最初原型之间的联系

        • 从上图可以看出，重写原型对象切断了现有原型与任何之前已经存在的对象实例中的联系，它们引用的仍然是最初的原型。

  • 原生对象的原型

    • 所有原生的引用类型，都是采用这种方式创建的。

  • 原型对象的问题

    • 省略了为构造函数传递初始化参数这一环节，结果所有实例在默认情况下都将取得相同的属性值。对包含引用类型值的属性来说，问题比较突出。

    • 实例一般都是要有属于自己的全部属性的，这个问题正是我们很少看到有人单独使用原型模式的原因所在。

  • 组合使用构造函数模式和原型模式

  • 动态原型模式

    只有在sayName()方法不存在的情况下，才会把它添加到原型中。

  • 寄生构造函数模式

    • 这个模式可以在特殊情况下用来为对象创建构造函数。假设我们想创建一个具有额外方法的特殊数组，但不能直接修改Array的构造函数，因此可以使用这个模式。
